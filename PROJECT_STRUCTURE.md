# 🧱 项目结构与代码规范

本文档说明 UltimateDESIGN 13阶段重构后的目录职责、文件命名、模块边界和底线维护规则。目标是让后续开发者能迅速建立上下文：**功能应该放在哪里、页面如何扩展、哪些底层引擎逻辑禁止随意修改**。

---

### 🗂️ 核心目录架构

```text
ultimateDESIGN/
├─ 🏠 app.py                       # 首页、平台状态、全局地图 2D/3D 基底入口
├─ 🧭 pages/                       # 核心视窗：文件名决定侧栏呈现与路由执行顺序
├─ 🧬 src/                         # 领域代码：核心逻辑与可复用套件
│  ├─ ⚙️ config/                   # 静态变量、路径注册、配置加载
│  ├─ 🧠 engines/                  # 计算、大模型、AIGC 逻辑层（严禁在此写 UI）
│  ├─ 🎛️ ui/                       # 全局外壳、原子设计系统、图表配色方案
│  ├─ 🧰 utils/                    # IO 操作、数据转换、文本清洗等工具函数
│  └─ 🧩 workflow/                 # 13 阶段工作流映射引擎与数据总线
├─ 🎨 assets/                      # 前端静态物料：CSS 样式、HTML WebGL 模板
├─ 🗃️ data/                        # 数据仓库：空间底座、表格、语义分析暂存文件
├─ 📄 docs/                        # 规划文书：任务书、开题报告、法定政策资料
├─ 🌐 static/                      # Streamlit Server 暴露的网络静态资源
├─ 🛠️ tools/                       # DevOps：环境检查与 Smoke 启动测试脚本
└─ ✅ tests/                       # 自动化验证：Pytest 单元测试
```

### 🧭 页面路由与 13 阶段映射 (pages/)

系统严格采用前缀编号管理页面顺序，分为前期、中期、后期三大设计板块：

#### 🔹 前期：数据获取与现状分析
- **`01_任务解读.py`**：透视项目概况、提取任务书要求，生成初始区位图提示词。
- **`02_资料收集.py`**：调用 MarkItDown 语义萃取文档内容，盘点空间数据资产。
- **`03_现场调研.py`**：检索并呈现四向街景照片，维护实地调研样本库。
- **`04_现状分析.py`**：加载 3D 底座，进行 POI 活力聚合、建筑形态与天际线统计。
- **`05_问题诊断.py`**：运算 AHP-MPI 评估模型，绘制地块诊断雷达并出具 AI 报告。

#### 🔹 中期：概念生成与应对策略
- **`06_目标定位.py`**：依托 LLM 推理对标相似案例，提炼并输出整体设计愿景。
- **`07_设计策略.py`**：模拟利益相关方（居民/开发商/规划师）的策略博弈与 RAG 政策约束检验。

#### 🔹 后期：设计生成与成果表达
- **`08_总体城市设计.py`**：向 Stable Diffusion 派发指令，进行概念总平面图 AIGC 生形推演。
- **`09_专项系统设计.py`**：构建并叠合交通、公共空间、风貌、历史文化四大专项的轴测分析体系。
- **`10_重点地段深化.py`**：选定 5 类微更新地块，进行 AIGC 街景前后推演比对 (Before/After)。
- **`11_实施路径.py`**：分类六大更新模式，排布并展示近中远三期实施时间轴与落地策略。
- **`12_城市设计导则.py`**：汇总 LLM 五阶段推演成果，确立空间管控指标并导出为规范化 Word 文档。
- **`13_成果表达.py`**：索引并管理全链条 41 个图纸提示词模板，作为展示与学术图册导出的总控中心。

### 🧬 领域代码与引擎职责 (src/)

核心逻辑已做严格解耦。**页面文件 (`pages/`) 只负责组装 UI，具体的运算与模型调度必须放置在对应的领域引擎中。**

- **`src/config/` (配置中枢)**
  一切新增的数据文件路径，必须优先在 `paths.py` 中登记注册。
- **`src/engines/` (神经中枢)**
  - `engine_registry.py`: 对外暴漏常用引擎能力，降低页面导入成本。
  - `spatial_engine.py`: 负责 POI 融合、街景处理、天际线运算与空间地块指标生成。
  - `site_diagnostic_engine.py`: 运行地块诊断算法与策略矩阵运算。
  - `stable_diffusion_engine.py`: 包装对 WebUI 的 RESTful 请求以执行图像推演。
  - `drawing_prompt_templates.py`: 集中维护 **41 个数据驱动的学术图纸提示词模板**。
  - `llm_engine.py / rag_engine.py`: 调度本地 Ollama (Gemma) 进行对话生成、提示词包装与本地库检索。
- **`src/ui/` (表现系统)**
  - `app_shell.py`: 控制全局导航与顶部 HUD 指示器，属于框架层能力。
  - `design_system.py`: 存放可复用的 Banner、分段卡片等视觉原子组件。
  - `module_summary.py`: 提供用于答辩的学术总结面板，必须被挂载在每个阶段页面的底部。
  - `chart_theme.py`: 集中定义 Plotly 的科技感配色与透明底样式，杜绝在页面内硬编码 UI。
- **`src/workflow/` (流程总线)**
  - `city_design_workflow.py`: 唯一维护 13 阶段信息与资源映射关系的地方。
  - `stage_data_bus.py`: 跨阶段数据总线。用于阶段间的证据链条传递。

### ⚡ 性能优化与渲染架构

系统在处理城市级大体量 GeoJSON（150公顷+建筑底座）时，采用了以下核心优化策略，后续二次开发时**务必遵守**：

- **统一数据流式静默缓存 (`@st.cache_data`)**
  对 `spatial_engine.py` 等处的高频耗时操作（如 IO 和运算）采用较长 TTL 的内存缓存机制。
  > [!WARNING]
  > **绝对禁止**在 `@st.cache_data` 中使用 `show_spinner=False` 参数，以防打破向下兼容导致旧版本 Streamlit 崩溃及 CI 测试挂起。
- **地图组件静态渲染与局部按需刷新 (`@st.fragment`)**
  首页 `render_project_map()` 强封装在 `@st.fragment` 装饰器下。当用户切换左侧交互控件（开关建筑轮廓、推演光照）时，**引擎仅会重绘该地图代码块**，绝不导致整个 Streamlit App 主线程重启。
  WebGL 框架 (`map3d_standalone.html`) 经字符串读取后传入 `components.html`，底层采用 JS Fetch 异步拉取巨型静态文件 (`.geojson`)，完全避免 Python 后台假死。
- **异步大模型推演**
  所有与本地 Gemma/SD 的握手及计算任务，原则上结合 Streamlit 状态机交由后台执行，确保 UI 线程的极速响应。

### 🛠️ 底线维护规则

1. **路由克制**：跨页面跳转一律交由 `app_shell` 中的顶部导航栏统一接管。
2. **逻辑下沉**：坚决避免在 `pages/` 内书写复杂的业务计算与模型调用，请移步 `src/engines/`。
3. **样式隔离**：页面级的 UI 优化应提炼为组件存入 `src/ui/design_system.py`，禁止在页面复制冗长的 CSS Markdown。
4. **验证优先**：任何核心逻辑的变动，提交前至少执行以下健康自检流程：

```powershell
# 编译全项目合规性
python -m compileall app.py pages src tests tools
# 跑通核心算法单元测试
pytest
# 挂载 Smoke 启动页探测脚本
python tools/startup_smoke.py
```
