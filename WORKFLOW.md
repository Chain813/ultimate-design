# 城市更新平台全流程工作流 (Urban Renewal Platform Workflow)

## 1. 导言与核心交付目标

本平台的核心使命是**面向图纸与视频生成的全流程自动化设计**。
项目最终交付的成果是一套**高精度的专业规划图册**与**项目推演动画视频**。
因此，虽然本平台涵盖了从数据收集、指标诊断到多主体博弈等诸多复杂的中间环节，但所有这些前置步骤（Stage 00 - Stage 07）都是在为后期的**图纸生成系统 (AIGC Drawing Pipeline)** 积攒严谨的“空间约束（ControlNet 底图）”与“逻辑依据（Prompt 提示词）”。

> [!IMPORTANT]
> **关于生图系统（图册与视频）的现状与未来**
> 目前，核心的生图系统与图册自动排版功能尚未完全完善。本文档旨在梳理清 14 个阶段（Stage 00 - Stage 13）的完整流转逻辑，详细说明**每个阶段在“第一步点什么、第二步点什么”的同时，是如何为最终的图册与视频生成“铺路”的**。后续系统的开发重心将全面向“高质量出图与视频渲染”倾斜。

## 2. 全流程 14 步标准操作指南

### 【前期：数据底座与现状诊断】

#### Stage 00: 数据准备
- **阶段目的**：上传项目所需的各类原始数据，建立物理与文本数据资产库。
- **操作流程**：
  1. 进入【数据准备 -> 数据上传中心】。
  2. 上传空间底图资料：包含研究范围红线（必需）、卫星正射影像、GIS专题图、路网矢量文件等。**注意**：所有矢量空间数据统一存放于 `data/gis/` 目录下；超大原始文件存放于 `data/gis/raw/`（已配置 `.gitignore` 避免仓库臃肿）。
  3. 上传分析数据：街景影像采样数据（用于品质诊断）、公共服务 POI、交通拥堵热力图等。统计表格类数据统一存放于 `data/csv/` 目录下。
  4. 上传文本资料：任务书、开题报告、社交媒体评论等。
- **为图册与视频提供的核心支撑**：
  - **绝不可缺少的空间约束**：此阶段上传的红线边界、路网和卫星底图，将作为后续 ControlNet 渲染的物理锁定层（Canny/Lineart）。这是保障 AI 出图能够被称为“标准城乡规划图纸”的最底线要求。

#### Stage 01: 任务解读
- **阶段目的**：从非结构化的规划任务书中明确项目的刚性需求和研究边界。
- **操作流程**：
  1. 进入【数据底座与规划策略 -> 策略语义萃取】。
  2. 挂载已上传的任务书与开题报告 PDF/Word。
  3. 启动“任务解读”，系统自动提取项目的名称、类型、面积、设计深度及核心考核指标。
  4. 在首页的【街区范围及改造红线】核对 2D/3D 基底是否正确映射。
- **为图册与视频提供的核心支撑**：
  - 为最终生成的图册图框、标题栏提供准确的项目名称和指标占位符，确保图面文字规范。

#### Stage 02: 资料收集
- **阶段目的**：构建后续自动化分析与设计推演的知识库。
- **操作流程**：
  1. 进入【数据底座与规划策略 -> 物理底座管理】，检查各个空间图层（用地、建筑、路网）的状态为“已接入”。
  2. 进入【策略语义萃取】，对政策法规、历史名城保护规划等文档进行批量 Markdown 转换与萃取。
- **为图册与视频提供的核心支撑**：
  - 沉淀《区位分析图》、《上位规划解读图》的必备文本分析素材。
  - 此阶段确认的“紫线保护范围”（历史建筑），将作为后续 AI 强制叠加（Alpha Composite）的核心遮罩图层，确保历史建筑在生图中绝对不可篡改。

#### Stage 03: 现场调研
- **阶段目的**：基于街景数据和人为标记，建立微观环境的视觉档案。
- **操作流程**：
  1. 进入【现场调研 -> 现场调研街景样本库】。
  2. 导入现场采集的四个方向的全景/街景切片。
  3. 结合打分脚本，对主要节点的绿视率、围合感进行基线测定。
- **为图册与视频提供的核心支撑**：
  - 此阶段的照片将直接作为最终《街区风貌修缮预演（Before/After）》和视频动画中推演镜头的原图底板（Init Image）。

#### Stage 04: 现状分析
- **阶段目的**：基于多源数据（POI、交通、街景），完成空间全景的量化诊断。
- **操作流程**：
  1. 进入【现状空间全景诊断 -> 3D现状全息底座】。
  2. 开启建筑底座和规划用地图层，检查肌理。
  3. 叠加社会活力图层（POI热力、交通拥堵），直观审视空间问题。
  4. 开启街景品质 3D 柱体，查看空间短板分布。
- **为图册与视频提供的核心支撑**：
  - 为自动生成《用地现状分析图》、《道路交通现状图》和各种热力图提供可视化的底图与截图参考，保证现状诊断图册的数据真实性。

#### Stage 05: 问题诊断
- **阶段目的**：通过测度模型锁定近期应该优先启动的“微更新地块”。
- **操作流程**：
  1. 进入【数据底座与规划策略 -> 资产综合评估】。
  2. 调节 AHP 权重滑块（空间潜力、社会需求等），计算地块的 MPI（更新潜力指数）。
  3. 进入【现状空间全景诊断 -> 地块级诊断面板】，查看地块排行，并导出 `Plot_Diagnostics_Report.csv`。
  4. 生成多维雷达图和问题清单报告。
- **为图册与视频提供的核心支撑**：
  - 支撑生成《四大问题诊断总图》、《更新潜力评价图》。
  - MPI 得分高的地块，将直接被送入“重点地段深化（Stage 10）”，进行高精度的风貌重塑，作为视频演示的“主舞台”。

---

### 【中期：智能博弈与概念生成】

#### Stage 06: 目标定位
- **阶段目的**：基于问题与政策依据，提炼项目愿景与设计理念。
- **操作流程**：
  1. 进入【LLM博弈决策 -> 阶段三：设计理念】。
  2. 整合任务书依据与前期诊断报告，让大模型生成“四大核心策略”。
- **为图册与视频提供的核心支撑**：
  - 为 AI 生图 Prompt 注入“设计灵魂”（如：是偏向高科技 TOD 还是偏向复古文化漫步），保障图面视觉气质统一。

#### Stage 07: 设计策略（多主体博弈）
- **阶段目的**：模拟城市更新中不同利益相关者的诉求，形成具备落地性的空间策略。
- **操作流程**：
  1. 进入【LLM博弈决策 -> 阶段二：方案借鉴】，提取对标案例经验。
  2. 进入【阶段四：问题-策略对应】，针对某一微更新提案启动“三角色协商（居民、开发商、规划师）”。
  3. 进行 RAG 政策合规预审，观察“动态共识雷达”。
- **为图册与视频提供的核心支撑**：
  - 多方妥协后的结论，将决定《功能策划图》和《更新模式分区图》中“拆、改、留”色块的具体比例分配，确保图纸不仅好看，更具备落地合理性。

---

### 【后期：核心图册与动画生图系统（待完善重心）】

#### Stage 08: 总体城市设计
- **阶段目的**：完成宏观层面的空间结构和总平面生成。
- **操作流程**：
  1. 进入【AIGC设计推演 -> 概念总平面图生形】。
  2. 配置生图约束：确保已挂载红线（Weight 1.0）、路网（Canny）和土地利用（Seg）。
  3. 执行生成，获取 2K 级别的总平面规划图。
- **为图册与视频提供的核心支撑**：
  - 直接产出整个图册的核心骨架主图：《总平面图》和《空间结构规划图》。

#### Stage 09: 专项系统设计
- **阶段目的**：拆解总图，生成专项规划系统图纸。
- **操作流程**：
  1. 结合前期分析的底图与大语言模型梳理的逻辑。
  2. 利用图纸提示词助手，按章节批量输出专项图纸。
  3. **智能容错与降级机制**：如果系统检测到缺少关键的空间资产（如红线图、路网图），将不再直接拒绝出图，而是自动降级切换至**“示意/概念表达模式”**，允许大模型补全占位符并完成视觉模板排版，确保流程不中断。
- **为图册与视频提供的核心支撑**：
  - 产出《道路交通系统图》、《公共空间网络图》、《历史文化展示系统图》。此时系统利用 ControlNet GIS 专题图层强制规范各类用地色块。

#### Stage 10: 重点地段深化
- **阶段目的**：对重点更新地块进行微观形态设计与风貌表达。
- **操作流程**：
  1. 进入【AIGC设计推演 -> 轴测鸟瞰空间体块模拟】进行 3D 体量推演。
  2. 进入【街区全景透视推演】，基于现状照片进行局部修缮与立面更新（Before/After）。
  3. 保存高质量图象至重点地块图集库。
- **为图册与视频提供的核心支撑**：
  - 直接产出高质量汇报所需的《重点地块效果图》、《街道断面图》和《人视场景图》。
  - **这里的 Before/After 对比图是构建最终“推演动画视频”不可或缺的关键帧素材。**

#### Stage 11: 实施路径
- **阶段目的**：将更新动作落实到时间轴和图层上。
- **操作流程**：
  1. 进入【更新设计成果展示 -> 更新设计大地图】。
  2. 综合前面的策略，生成严格的“留改拆”地图。
- **为图册与视频提供的核心支撑**：
  - 产出《实施分期图》和《留改拆总图》。在这个环节，底层系统会强制应用“历史建筑保护遮罩”的像素级 Overlay，确保图纸上“留”的部分绝对不可篡改。

#### Stage 12: 城市设计导则
- **阶段目的**：将空间策略沉淀为可指导施工的文本规范。
- **操作流程**：
  1. 进入【LLM博弈决策 -> 空间成果方案】。
  2. LLM 自动将图纸和博弈结论翻译为高度、强度、色彩、退让的控制条文。
- **为图册与视频提供的核心支撑**：
  - 为《风貌控制图》和《建筑更新控制图》提供精确的表格数据排版内容。

#### Stage 13: 成果表达 (终极交付环节)
- **阶段目的**：全流程所有心血的最后汇聚，打包图册与视频。
- **操作流程**：
  1. 进入【LLM博弈决策 -> 图纸提示词助手】统筹整个画廊库。
  2. 导出包含规划文本、导则和说明文件的 Word/Markdown。
  3. **(待完善)** 启动一键排版，将画廊库图纸组装为标准图册。
  4. **(待完善)** 提取核心图纸帧，一键生成项目推演动画视频。
- **交付终态**：
  - 按照《01项目认知篇》到《07推演与实施篇》逻辑组装的专业级 **A3/A1 规划图册**，以及配套的推演汇报**视频短片**。

## 3. 项目架构与完整性保障 (Project Architecture & Integrity)

为了保障全流程的稳定运行与零运行时错误，本平台在底层架构上实施了严格的完整性规范：

- **数据资产清晰分层**：所有的空间矢量数据集中在 `data/gis/`，表格型指标数据集中在 `data/csv/`，并在 `src/config/paths.py` 中实现了全局路径注册。前端展示用数据（如 Web 3D 地图资产）单独隔离在 `static/` 目录下。
- **自动化诊断哨兵**：每次启动或更新数据前，可通过内置的 `tools/data_quality_check.py` 生成完整的数据质量检测报告，并利用 `tools/startup_smoke.py` 进行页面路由冒烟测试，拦截任何因路径错位引起的崩溃。
- **Git 大文件管控**：通过完善的 `.gitignore` 规则，将 `data/gis/raw/` 下的 GB 级别原始路网、铁网 JSON 拦截在 Git 索引之外，保障了团队 GitHub 协同效率与云端仓库健康。
